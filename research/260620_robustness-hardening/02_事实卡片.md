# 事实卡片

## F1：Event Tap 指针存在不等于监听健康

- 层级：官方机制 + 多项目实现
- 事实：Core Graphics 同时提供 Mach port 有效性和 tap enabled 状态；Hammerspoon 与 LinearMouse 都用真实状态判断，而不是只判断引用非空。
- 当前映射：SafariGestures 的 `isRunning` 仅为 `tap != nil`。
- 结论边界：可确认当前健康判断不充分；不能据此断言现网已经失效过。

## F2：tap 可能被超时禁用，也可能静默失效

- 层级：官方机制 + LinearMouse 修复记录
- 事实：Apple SDK 说明 tap 无响应时会被禁用；LinearMouse 2026-04 因无效返回事件被系统静默禁用，加入了 5 秒健康检查。
- 当前映射：SafariGestures callback 做 UI 路径计算，长轨迹开销会增加 timeout 风险。
- 结论：需要低频健康检查和完整重建路径，但不需要额外进程级 watchdog。

## F3：睡眠和用户会话切换应被视为生命周期边界

- 层级：Apple 通知 API + LinearMouse 当前源码
- 事实：Apple 提供 sleep/wake 和 session active/inactive 通知；LinearMouse 在 sleep / inactive 时 stop，在 wake / active 时重新 start。
- 当前映射：SafariGestures 没有注册这些通知。
- 结论：纳入第 6 步，采用 stop/rebuild，而不是只在唤醒后调用一次 enable。

## F4：手势触发键必须使用显式状态协调

- 层级：活跃项目 PR + issue
- 事实：LinearMouse #1138 出现 trigger button 未释放；PR #1246 用“按下时延迟 fallback，释放时根据是否命中手势决定补发 click”的纯状态协调器修复相关路径。
- 当前映射：SafariGestures 已采用类似扣住 down、up 时决策的结构，但状态散落在 `isTracking`、`points` 和 overlay 中。
- 结论：抽出单一会话状态机并测试所有取消路径，必要且直接相关。

## F5：测试本身可以把系统鼠标状态卡住

- 层级：LinearMouse 已合并修复
- 事实：LinearMouse 单元测试曾向真实 event tap 发送 mouseDown 而没有配对 up，导致 WindowServer 认为按钮持续按下；修复后改为依赖注入的记录器，真实系统事件测试默认跳过。
- 当前映射：SafariGestures 计划新增状态机测试。
- 结论：普通 `swift test` 必须纯逻辑、零系统事件；真实事件只允许人工或显式 opt-in 集成测试。

## F6：后台 Event Tap 线程不是免费加固

- 层级：LinearMouse / Mac Mouse Fix 源码与修复记录
- 事实：LinearMouse 为高吞吐滚轮处理迁入专用线程后，又修复多处 AppKit 线程崩溃和状态竞态；Mac Mouse Fix 也记录过线程切换导致 click-and-drag 卡死。
- 当前映射：SafariGestures 只监听右键会话，静置 3 线程、CPU 0%，处理量远低于两者。
- 结论：不引入专用事件线程；先消除 O(n²) 轨迹计算。

## F7：轨迹数组和路径长度存在确定的复杂度风险

- 层级：本地源码静态分析
- 事实：每个 dragged 事件都会追加点；`pathLength()` 每次追加都遍历全部点；数组无上限且结束时保留容量。
- 当前映射：高轮询率鼠标和异常长按会放大 CPU、内存高水位和 tap timeout 风险。
- 结论：增量路径长度、距离降采样、硬上限和超时取消必须纳入。

## F8：多屏 overlay 不能固定取第一块屏幕

- 层级：本地源码 + MacGesture / Hammerspoon issue + Apple 通知 API
- 事实：SafariGestures 固定使用 `NSScreen.screens.first`；MacGesture #90 和 Hammerspoon #3263 都出现外接屏/垂直偏移错位；Apple 提供屏幕参数变化通知。
- 结论：按手势起点选择屏幕，监听配置变化，并在变化时取消当前会话。

## F9：`CGShieldingWindowLevel` 有残影风险

- 层级：同源实现 + 真实 issue
- 事实：SafariGestures 与 MacGesture 都使用 shielding level；MacGesture #139 在 Mission Control / 空间切换出现覆盖层残影。LinearMouse 的点击穿透提示层使用 `.statusBar` + `.fullScreenAuxiliary`。
- 结论：SafariGestures 应降到 `.statusBar`，并做 Mission Control / 全屏回归。

## F10：运行中撤销辅助功能权限是危险场景

- 层级：LinearMouse #667、Mac Mouse Fix #346
- 事实：两项目均报告运行中撤权后系统点击失效；LinearMouse 使用额外 test tap watchdog，失败时重启应用规避。
- 当前映射：SafariGestures 只拦截右键，不修改设备驱动，风险低于它们，但仍可能在 tracking 中断时残留内部状态。
- 结论：不复刻“每 10 秒创建 test tap + 重启整个 App”；只在 disabled / health check 失败时停止监听并显示缺权限，测试撤权前必须先停用 App。

## F11：当前快捷键没有已确认的修饰键残留

- 层级：本机实测 + 外部风险案例
- 事实：当前运行版多次执行 Cmd/Shift 快捷键后，combinedSessionState 与 hidSystemState flags 均为 0；未观察到残留。LinearMouse 和 Hammerspoon 有其它合成路径导致修饰键残留的案例。
- 结论：加入专项人工验证和 opt-in 集成测试；在复现前不重写 KeySender，不额外引入 flagsChanged 序列。

## F12：当前短时资源表现良好，但不能证明 24 小时稳定

- 层级：本机实测
- 事实：约 35 分钟运行中，静置 CPU 0%、物理内存 18.3MB、峰值 19.7MB、leaks 报告 0；没有当前版本 crash report。
- 结论：不存在立即可见泄漏；第 6 步应做压力和长稳验证，而不是假定已有泄漏后大改所有权结构。

## F13：`SMAppService.mainApp` 不提供崩溃自动拉起

- 层级：当前 SDK header
- 事实：main app 注册后在后续登录时启动；“非零退出后重启”只写在 LoginItem helper 的行为中。LaunchAgent 可配置 KeepAlive，但需要额外架构。
- 当前映射：当前 App 没有崩溃记录，且菜单提供正常退出。
- 结论：本轮不加 LaunchAgent；若长稳测试出现真实崩溃，再评估 `Crashed=true` 或 `SuccessfulExit=false`，不能用无条件 KeepAlive。

## F14：签名私钥 ACL 是独立安全阻塞项

- 层级：本机 `security` 官方帮助
- 事实：脚本使用的 `-A` 被系统标记为不安全；该私钥定义稳定签名身份并关联辅助功能授权。
- 结论：正式安装前必须移除 `-A`、限制 codesign 使用，并处理已导入旧私钥；这是安全收尾，不是 Event Tap 稳定性功能。

# Safari 鼠标手势工具 — 设计与实施计划

> 目标:做一个轻量的 macOS 菜单栏小程序,在 Safari 里用「按住右键划动」触发前进/后退/标签操作等。
> 替代 WGestures(系统级、内存偏高),只管 Safari、更轻。

## 〇、调研结论:为什么不用现成的(已验证)

动手前评估过现成方案,结论是它们有对用户致命的硬伤,所以决定自建:

| 方案 | 结论 |
|---|---|
| **Mouse Gestures for Safari**(App Store 免费扩展) | 已实测安装。9 个手势里只支持 7 个(**缺「重开关闭的标签」「停止加载」**);更致命的是 **右键单击后,普通右键菜单要改成 Ctrl+点击 / 双击右键才能调出**——见下方原理,这是扩展架构的天生限制,无法绕过。 |
| **MacGesture**(开源 GPL) | 跟本项目同类,但最后更新 2023-01,在 macOS 15 / Safari 18.5 上兼容性存疑;且仍是系统级,不一定比 WGestures 轻。 |
| **WGestures**(用户原方案) | 系统级、内存偏高,是本项目要替代的对象。 |

**为什么扩展做不到「单击照常出菜单」,而自建能做到(核心设计点):**
- WebKit 的规矩是「右键**一按下**,菜单立刻弹」,不给扩展「等一下看你动不动」的机会。扩展(跑在网页里)只能把菜单全禁掉,再用 Ctrl+点击当暗号 → 所以单击出菜单要按两下。
- 自建的原生工具站在**系统事件层**(CGEventTap),比 Safari 更早拿到鼠标事件,能**等到松开右键再决定**:没移动就把右键原样还给 Safari(菜单一次单击正常弹),移动了就吃掉并执行手势。**这正是 Edge / 旧版 WGestures 的做法,是本工具相对扩展的核心价值,必须实现且作为一级验收项。**

## 一、最终形态

- **菜单栏 App**(`SafariGestures.app`):无窗口、无 Dock 图标(`LSUIElement=true`),只在菜单栏挂一个图标,提供「开关 / 退出 / 关于」菜单。
- **从源码命令行编译**:用本机已有的 Swift 6.1.2 + Command Line Tools,**不需要完整 Xcode、不需要 Apple 开发者账号、不需要代码签名**。
- **权限**:首次运行需用户在「系统设置 → 隐私与安全性 → 辅助功能」里勾选授权(所有全局鼠标工具的通用门槛)。
- **开机自启**:用 `SMAppService`(macOS 13+)注册登录项,或放进登录项。
- **预期内存**:10–20MB 级别。

## 二、技术方案

### 核心技术
- **事件监听**:`CGEvent.tapCreate`(Quartz Event Services),监听 `rightMouseDown / rightMouseDragged / rightMouseUp`。
- **作用域限定**:仅当最前台 App 是 Safari(`NSWorkspace.shared.frontmostApplication?.bundleIdentifier == "com.apple.Safari"`)时才拦截手势;否则原样放行,不影响其他程序的右键。
- **发送快捷键**:`CGEvent(keyboardEventSource:virtualKey:keyDown:)` 合成按键 + 修饰键(Cmd/Shift/Option),`post` 给系统。

### ★ 一级需求:区分「画手势」 vs 「普通右键点一下」(成败核心)
**这是本工具相对免费扩展的核心价值,必须做对——验收时一票否决项。**
目标行为 = 跟 Edge 完全一致:**右键单击一次 → 菜单照常弹;右键按住划动 → 执行手势、不弹菜单。**

1. `rightMouseDown` 时:开始记录轨迹点,**先扣住事件不放行**(这样菜单不会立刻弹)。
2. `rightMouseDragged` 时:持续记录点。
3. `rightMouseUp` 时:
   - 若移动距离 **超过阈值**(建议 ~30px,可调)且识别出有效手势 → **吃掉这次右键事件**(return nil,不让菜单弹),发送对应快捷键。
   - 若几乎没移动(就是想右键点一下) → **原样补发右键 down+up 给 Safari**,让原生右键菜单**一次单击就正常弹出**(绝不能要求用户 Ctrl+点击或双击)。
4. 边界:扣住事件不能造成右键卡死/丢失;补发的事件位置、按键状态要正确,菜单弹出位置正常。WGestures / MacGesture 已验证此路可行,可参考其思路。

### 手势识别算法(可单元测试的纯函数)
输入:轨迹点数组 `[(x,y)...]`;输出:方向序列字符串(如 `"DR"` = 先下再右)。
- 设最小段长阈值(如 20–30px),过滤抖动。
- 逐段判断主方向(比较 |dx| 与 |dy|)归类为 L/R/U/D。
- 合并相邻相同方向,得到方向序列。
- 查手势表匹配 → 得到动作。
- **这一段必须抽成独立纯函数 + 配单元测试**(见质检部分)。

### 手势 → 快捷键映射表(最终 9 个)

| 方向序列 | 动作 | 快捷键 | 虚拟键码(参考) |
|---|---|---|---|
| `L` | 返回 | Cmd+[ | 33 + Cmd |
| `R` | 前进 | Cmd+] | 30 + Cmd |
| `DR` | 关闭标签页 | Cmd+W | 13 + Cmd |
| `LU` | 重开已关标签 | Cmd+Shift+T | 17 + Cmd+Shift |
| `RU` | 新建标签页 | Cmd+T | 17 + Cmd |
| `RD` | 刷新 | Cmd+R | 15 + Cmd |
| `UL` | 切到左边标签 | Cmd+Shift+[ | 33 + Cmd+Shift |
| `UR` | 切到右边标签 | Cmd+Shift+] | 30 + Cmd+Shift |
| `DL` | 停止加载 | Esc | 53 |

> 键码 Codex 实现时以 `Carbon/HIToolbox` 或实测为准,本表仅供对照。
> 映射建议抽成一份配置(代码内常量表即可),方便日后改。

## 三、工程结构与编译

建议用 **Swift Package Manager**(纯命令行,无需 Xcode GUI):

```
safari-gestures/
├── PLAN.md                  # 本文档
├── Package.swift            # SPM 配置(可执行 target)
├── Sources/SafariGestures/
│   ├── main.swift           # 入口,菜单栏 App
│   ├── AppDelegate.swift     # 菜单栏图标 + 菜单 + 权限检查
│   ├── EventTap.swift        # CGEventTap 监听右键
│   ├── GestureRecognizer.swift  # 轨迹 → 方向序列(纯函数,可测)
│   ├── GestureMap.swift      # 方向序列 → 动作 → 快捷键
│   └── KeySender.swift       # 合成并发送快捷键
├── Tests/                   # 单元测试(测识别算法)
└── scripts/
    ├── build.sh             # swift build -c release
    └── make-app.sh          # 把可执行文件打包成 .app bundle(写 Info.plist)
```

**Info.plist 关键项**:`LSUIElement=true`(无 Dock 图标)、`NSAccessibilityUsageDescription`(权限说明)。

## 四、分步实施(Codex 执行)

> 每步做完交付一次,Claude 审一步再放行下一步,避免一口气写完难排查。

- **第 1 步**:搭骨架——SPM 工程 + 菜单栏图标 + 退出菜单 + 检测是否已授权辅助功能(没授权就引导)。能编译出 `.app` 并在菜单栏看到图标即可。
- **第 2 步**:CGEventTap 监听右键,加日志:在 Safari 里按住右键划动,控制台能打印出识别到的方向序列。**先不发快捷键、不拦截菜单**,纯观察识别准不准。
- **第 3 步**:接入手势表 + 发送快捷键。9 个手势逐个调通。
- **第 4 步**:处理「画手势 vs 普通右键」——加阈值,普通右键放行菜单,手势吃掉菜单。
- **第 5 步**:菜单栏加「启用/停用」开关 + 开机自启;打包脚本完善;写单元测试。

## 五、质检 / 测试方案(Claude 验收)

### 自动化(能测的部分)
- **单元测试**:针对 `GestureRecognizer` 纯函数,喂模拟轨迹点,断言输出方向序列正确(含抖动过滤、斜线归类、短距离判定为「非手势」)。Codex 必须随代码交付测试,`swift test` 全绿。

### 手动验收清单(在 Safari 真机逐条过)
- [ ] 9 个手势逐一触发,动作正确(返回/前进/关标签/重开/新建/刷新/切左/切右/停止)
- [ ] **★ 普通右键单击一次** → Safari 原生右键菜单照常弹出(**不需要 Ctrl+点击、不需要双击**,这是一票否决项)
- [ ] **非 Safari 程序**(如访达、备忘录)里右键和划动:完全不受影响
- [ ] 手势识别**不误触**:慢速、斜着划不乱发命令
- [ ] **不重复触发**:一次手势只发一次快捷键
- [ ] 菜单栏「停用」后手势失效,「启用」后恢复
- [ ] 重启电脑后自动运行,权限仍在
- [ ] 挂一段时间后内存稳定(无泄漏),CPU 空闲时接近 0

### 安全 / 隐私
- 工具只读鼠标事件 + 看最前台 App 包名,**不读键盘内容、不联网、不存任何数据**。审代码时确认无网络请求、无文件写入(日志可选且默认关)。

## 六、项目治理：测试版 vs 正式版（环境隔离 / 发布管理）

- **当前 `files/safari-gestures/` = 测试/开发版**：用的都是临时测试案例 + ad-hoc 临时签名，随便折腾。
- **全部功能完成 + 验收通过后 → 产出正式版**：稳定签名、装到正式位置（如 `~/Applications/SafariGestures.app`）、开机自启。
- **之后任何加功能 / 改 bug**：先在测试版改 → 测试 → 验收 → 再同步到正式版发布。**不直接动正式版**。

## 七、第 6 步：健壮性加固 / 生产就绪审查（功能全做完后）

> 工程术语：**可靠性加固 / 生产就绪审查（production-readiness review）/ 非功能性需求（NFR）**——除了"功能对"，还要"长期跑得稳、不泄漏、不被杀、异常不崩"。

做法：① 按经验列 + ② **借鉴同类开源（MacGesture / Mac Mouse Fix 等）的已知坑与加固手法**，提前写好防范，不等出事再救。

加固清单（待补全 + 待扒开源补充）：
- [ ] 内存 / CF 对象泄漏：`leaks` / Instruments 跑一遍，确认 event tap、RunLoop source、overlay 窗口无泄漏
- [ ] 长时间运行（挂 24h）：CPU 空闲≈0、内存平稳不增长
- [ ] Event Tap 被系统禁用后自愈（已有 tapDisabled 重启，需压测验证）
- [ ] 睡眠唤醒 / 切换用户 / 改分辨率·多屏后，tap 与轨迹层仍正常
- [ ] 被系统杀掉后能自动拉起（LaunchAgent KeepAlive 或登录项）
- [ ] 右键永不卡死：up 事件丢失等异常路径有兜底
- [ ] 异常 / 极端输入不崩溃（防御式编程）
- [ ] 资源释放干净（退出、停用时）
- [ ] 隐私复核：无网络、无文件写入、不读键盘内容
